# Inspection Management API

![CI](https://github.com/tiagorcfortunato/inspection-management-api/actions/workflows/ci.yml/badge.svg)

A backend REST API for managing infrastructure inspections вЂ” road damage reports, AI-assisted severity classification, status tracking, filtering, pagination, and sorting.

Built to demonstrate a production-grade, AI-native backend architecture: FastAPI, SQLAlchemy, PostgreSQL, Alembic, Docker, LangChain with structured LLM output, and full observability via LangSmith.

## Tech Stack

- **Python** + **FastAPI** вЂ” async web framework with dependency injection
- **SQLAlchemy** вЂ” ORM
- **PostgreSQL** вЂ” database
- **Alembic** вЂ” database migrations
- **Docker** вЂ” containerised execution
- **LangChain** + **ChatOpenAI** вЂ” LLM orchestration with structured output enforcement
- **LangSmith** вЂ” end-to-end LLM tracing and evaluation
- **Pydantic** + **Pydantic-Settings** вЂ” type-safe data validation and environment configuration
- **Pytest** вЂ” 31 automated tests covering auth, CRUD, filtering, pagination, sorting, validation, and admin access control

## Features

### Authentication
- User registration and login
- JWT-based authentication
- Protected routes via dependency injection

### Inspection Management
- Create, retrieve, update, and delete inspection records
- Status lifecycle: `reported` в†’ `verified` в†’ `scheduled` в†’ `repaired`
- Each user only has access to their own inspections

### Filtering
- Filter by `severity`, `status`, and `damage_type`

### Pagination
- `limit` and `offset` query parameters

### Sorting
- Sort by `reported_at`, `severity`, or `status`
- `asc` or `desc` order

## Elite AI Workflow Architecture

The API implements an autonomous, AI-native enrichment pipeline that runs behind the standard request lifecycle.

### Autonomous Background Pipeline

Inspection creation is non-blocking. On `POST /inspections`, the HTTP response is returned immediately with a `201 Created`. A `BackgroundTask` is dispatched concurrently to call the LLM, then writes the enriched classification back to the database record вЂ” zero added latency for the client.

```
POST /inspections
       в”‚
       в”њв”Ђв–є create_inspection()  в”Ђв”Ђв–є 201 response (instant)
       в”‚
       в””в”Ђв–є BackgroundTask: process_inspection_with_ai()
                  в”‚
                  в”њв”Ђв–є AIService.classify_inspection(notes)
                  в”‚         в””в”Ђв–є ChatOpenAI в†’ structured AIClassification
                  в”‚
                  в””в”Ђв–є UPDATE inspection SET
                            damage_type     = AI result,
                            severity        = AI result,
                            ai_rationale    = trust sentence,
                            is_ai_processed = true
```

### Explainable AI (XAI)

Every AI classification includes a `rationale` field вЂ” a single sentence generated by the model explaining its decision. This is persisted as `ai_rationale` and exposed in the API response, giving human domain experts a transparent audit trail to validate or override the model's output rather than treating it as a black box.

### Production-Grade Observability

LangSmith tracing is wired directly into the LLM client via environment config (`LANGSMITH_TRACING`, `LANGSMITH_API_KEY`, `LANGSMITH_PROJECT`). Every LLM call вЂ” input prompt, output tokens, latency, structured result вЂ” is captured as a traceable run, enabling regression detection, prompt evaluation, and cost monitoring without any code instrumentation.

### Data Integrity

`ChatOpenAI` is bound to `AIClassification` via LangChain's `.with_structured_output()`. The model is constrained to return exactly the `DamageType` and `SeverityLevel` enum values defined in the database schema. Invalid or hallucinated enum values are rejected at the LangChain layer before they can reach the ORM вЂ” making the LLM integration as type-safe as any internal function call.

## Architecture

The project follows a layered architecture where each layer has a single responsibility:

| Layer | Responsibility |
|---|---|
| **Routers** | Handle HTTP requests and responses |
| **Services** | Contain business logic |
| **Models** | Define database entities |
| **Schemas** | Validate request and response data |
| **Core** | Centralise auth, dependencies, and enums |

## Local Development

### 1. Set up environment variables

```bash
cp .env.example .env
```

Edit `.env` and fill in your values:

```
DATABASE_URL=postgresql://postgres:postgres@localhost:5433/taskdb
SECRET_KEY=your-secret-key-here
```

### 2. Install dependencies

```bash
pip install -r requirements.txt
```

### 3. Apply migrations

```bash
alembic upgrade head
```

### 4. Run the API

```bash
uvicorn app.main:app --reload
```

API available at `http://localhost:8000`
Swagger docs at `http://localhost:8000/docs`
